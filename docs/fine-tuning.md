## 1. Зачем дообучать

В демо детекция работает по **ground-truth манифесту** — система знает, где
дефект, потому что он туда нарисован (см.
[«Демо vs реальное производство»](../README.md#-демо-vs-реальное-производство)).
В реальном цехе ответа заранее нет: модель должна **сама** отличить настоящий
сгусток крови, гематому или костный фрагмент от:

- естественной текстуры и цвета мяса (мрамор, прожилки, кровь на срезе),
- бликов, теней, капель воды, льда,
- оборудования, спецодежды, конвейерной ленты.

Базовая VLM «из коробки» (`google/gemma-4-12b-qat`) понимает картинку в целом,
но не откалибрована под конкретный продукт, ракурс камеры и определение «дефекта»
вашего ОТК. Дообучение нужно, чтобы поднять **recall** (не пропускать брак) при
контролируемом **false-positive rate** (не стопорить линию зря).

## 2. Три стратегии

| Стратегия | Что делаем | Когда выбирать | Данных нужно |
|---|---|---|---|
| **A. Few-Shot / RAG** | Подкладываем модели эталонные примеры в промпт. Без обучения весов. | Быстрый старт, мало данных, проверка гипотезы. | 3–50 размеченных кадров |
| **B. Fine-tuning VLM** | Дообучаем веса Gemma (LoRA/QLoRA) под вашу задачу и формат ответа. | Нужна стабильность + объяснимость + bbox в JSON. | 500–5000+ кадров |
| **C. YOLO / RT-DETR** | Обучаем специализированный детектор объектов. | Нужна максимальная скорость и точность bbox на конвейере. | 1000–10000+ боксов |

На практике стратегии комбинируют: **C** ловит дефект быстро (мс на кадр), **B/A**
даёт текстовое объяснение и классификацию типа дефекта для отчёта ОТК.

## 3. Сбор и разметка данных

### 3.1 Источник кадров

Снимаем реальный поток с той же камеры и того же ракурса, что и в проде —
**доменный сдвиг убивает качество**. Модель, обученная на стоковом видео,
деградирует на вашей линии (другой свет, разрешение, motion-blur).

### 3.2 Разметка на лету (уже в проекте)

В `main.py` встроены горячие клавиши (см.
[`src/dataset_collector.py`](../src/dataset_collector.py)):

| Клавиша | Действие | Папка |
|---|---|---|
| `s` | сохранить **чистый** эталонный кадр | `dataset/positive/` |
| `d` | сохранить кадр с **дефектом** | `dataset/negative/` |

> ⚠️ Для обучения детектора (bbox) одной метки «дефект/чистый» мало — нужны
> **координаты** дефекта. Помечайте дефект рамкой во внешнем инструменте:
> [Label Studio](https://labelstud.io/), [CVAT](https://www.cvat.ai/) или
> [Roboflow](https://roboflow.com/).

### 3.3 Форматы аннотаций

- **VLM (стратегия B):** `image → JSON` с описанием и боксом, например:
  ```json
  {"defect": true, "type": "blood_clot", "confidence": 0.96,
   "boxes": [[ymin, xmin, ymax, xmax]]}
  ```
  Координаты — в шкале `0–1000` (конвенция Gemma) или нормализованные `0–1`.
- **YOLO (стратегия C):** на каждое изображение `.txt`:
  `class_id  x_center  y_center  width  height` (всё нормализовано `0–1`).
- **COCO:** единый `annotations.json` — поддерживается большинством фреймворков.

### 3.4 Сколько и какие данные

- **Баланс классов:** не только дефекты — много «трудных» чистых кадров
  (кровь на срезе, тени), иначе вырастет false-positive.
- **Разнообразие:** разный свет, скорость ленты, ракурсы, степень motion-blur.
- **Гигиена сплита:** train / val / test **по разным сменам/тушам**, чтобы
  соседние кадры одного эпизода не утекли из train в test (data leakage).
- **Минимум для старта:** ~300–500 размеченных дефектов на тип; промышленное
  качество — тысячи.

## 4. Подготовка датасета

В проекте есть заготовка [`prepare_training_json.py`](../prepare_training_json.py)
— собирает `dataset/positive` и `dataset/negative` в
`dataset/metadata.jsonl` формата HuggingFace (`{"file_name", "text"}`).

```bash
python prepare_training_json.py
```

Для **детекции с боксами** расширьте запись: добавьте поле с координатами и тип
дефекта (а не только текст). Итоговая структура:

```
dataset/
├── positive/                 # чистые эталоны (.jpg)
├── negative/                 # дефектные (.jpg)
├── annotations/              # боксы (Label Studio / CVAT экспорт)
├── metadata.jsonl            # для VLM (HuggingFace datasets)
└── yolo/                     # images/ + labels/ для YOLO
    ├── images/{train,val,test}/
    └── labels/{train,val,test}/
```

## 5. Стратегия A — Few-Shot / RAG

Уже реализована в [`src/analyzer.py`](../src/analyzer.py): при наличии ≥3 образцов
в `positive`/`negative` включается **Few-Shot Learning Mode** — эталонные
изображения подмешиваются в промпт, модель сравнивает текущий кадр с ними.

**Как улучшить без обучения весов:**

1. Накопите 10–30 разнообразных эталонов (`s`/`d`).
2. Подбирайте релевантные примеры динамически (RAG): по близости эмбеддингов
   кадра — подкладывайте 2–4 самых похожих эталона, а не случайные.
3. Ужесточите системный промпт: точное определение «дефекта» вашего ОТК,
   явные контрпримеры (кровь на срезе ≠ дефект).

Плюс: ноль обучения, мгновенно. Минус: дороже инференс (примеры в контексте),
потолок точности ниже, чем у fine-tuning.

## 6. Стратегия B — Fine-tuning VLM (LoRA/QLoRA)

Дообучаем **Gemma 4** под задачу. Полное дообучение 12B требует дорогих GPU,
поэтому используем **параметр-эффективный** подход — **LoRA/QLoRA** (обучаем
~0.5–2% весов, помещается на одну 24 ГБ видеокарту в 4-bit).

### 6.1 Окружение

```bash
pip install torch transformers peft trl bitsandbytes accelerate datasets
# Проще через готовые фреймворки:
#   - Unsloth        (быстрый LoRA, экономия VRAM)
#   - LLaMA-Factory  (конфиг-YAML, без кода)
```

### 6.2 Формат обучающего примера

Каждый сэмпл — `(изображение, инструкция, целевой JSON-ответ)`:

```json
{
  "image": "negative/frame_0421.jpg",
  "prompt": "Inspect the meat. Report defects as JSON.",
  "response": "{\"defect\": true, \"type\": \"blood_clot\", \"confidence\": 0.96, \"boxes\": [[210, 340, 430, 560]]}"
}
```

Обучаем модель воспроизводить **строго валидный JSON** — тот же контракт, что
парсит `analyzer.py` (`response_format=json_schema`).

### 6.3 Ключевые гиперпараметры (отправная точка)

| Параметр | Значение | Комментарий |
|---|---|---|
| метод | QLoRA, 4-bit (nf4) | влезает в 24 ГБ VRAM |
| LoRA rank `r` | 16–32 | больше rank → больше ёмкость |
| `lora_alpha` | 32–64 | обычно `2×r` |
| target modules | q/k/v/o + MLP проекции | + проектор зрения для bbox |
| learning rate | 1e-4 … 2e-4 | с косинусным затуханием |
| epochs | 2–4 | следить за переобучением по val |
| batch (effective) | 8–16 | через gradient accumulation |
| заморозка | vision encoder часто **замораживают**, учат LoRA на LLM + проекторе | |

### 6.4 Запуск (схематично, HuggingFace + PEFT/TRL)

```python
from transformers import AutoModelForImageTextToText, AutoProcessor, BitsAndBytesConfig
from peft import LoraConfig
from trl import SFTTrainer, SFTConfig

bnb = BitsAndBytesConfig(load_in_4bit=True, bnb_4bit_quant_type="nf4",
                         bnb_4bit_compute_dtype="bfloat16")
model = AutoModelForImageTextToText.from_pretrained(
    "google/gemma-4-12b-qat", quantization_config=bnb, device_map="auto")
proc = AutoProcessor.from_pretrained("google/gemma-4-12b-qat")

lora = LoraConfig(r=16, lora_alpha=32, lora_dropout=0.05,
                  target_modules="all-linear", task_type="CAUSAL_LM")

trainer = SFTTrainer(
    model=model, peft_config=lora, processing_class=proc,
    train_dataset=ds_train, eval_dataset=ds_val,
    args=SFTConfig(output_dir="out", per_device_train_batch_size=2,
                   gradient_accumulation_steps=8, learning_rate=2e-4,
                   num_train_epochs=3, bf16=True, logging_steps=10,
                   eval_strategy="steps", eval_steps=100, save_steps=200),
)
trainer.train()
trainer.save_model("gemma-4-12b-meat-lora")
```

### 6.5 Конвертация для LM Studio

LM Studio запускает **GGUF**. После обучения:

1. Слить LoRA с базой: `peft → merge_and_unload()`.
2. Сконвертировать в GGUF (`llama.cpp/convert_hf_to_gguf.py`) и квантовать
   (`Q4_K_M` / `Q5_K_M`).
3. Положить в папку моделей LM Studio, загрузить, активировать Vision Adapter.

## 7. Стратегия C — Специализированный детектор (YOLO)

Если нужна **скорость конвейера** (десятки–сотни кадров/с) и точные боксы —
обучите детектор объектов. VLM избыточна для «найти красное пятно».

```bash
pip install ultralytics
```

```yaml
# meat.yaml
path: dataset/yolo
train: images/train
val: images/val
names:
  0: blood_clot
  1: bruise
  2: bone_fragment
```

```bash
yolo detect train model=yolo11m.pt data=meat.yaml epochs=100 imgsz=1280 batch=16
yolo detect val   model=runs/detect/train/weights/best.pt data=meat.yaml
```

Плюс: единицы мс/кадр на GPU, высокий mAP на узкой задаче. Минус: только боксы,
без текстового объяснения (можно связкой YOLO → VLM для отчёта).

## 8. Оценка качества

Меряем **на отложенном test-сплите** (другие смены/туши):

- **Recall (полнота)** — доля пойманного брака. Главная метрика для пищевой
  безопасности: пропуск дефекта дороже ложной тревоги.
- **Precision (точность)** — доля верных тревог. Низкая → линия стоит зря.
- **mAP@0.5 / mAP@0.5:0.95** — качество боксов (для детектора).
- **Confusion matrix** по типам дефектов.
- **Latency** — мс/кадр на целевом железе.

Подбирайте **порог уверенности** по ROC/PR-кривой под бизнес-баланс
recall ↔ precision (для ОТК обычно смещают в сторону recall).

## 9. Возврат модели в систему

После обучения отключаем «демо-костыли» и включаем реальный инференс:

1. **Убрать манифест.** В `main.py` детекция идёт через
   [`src/defect_manifest.py`](../src/defect_manifest.py) — заменить вызов
   `manifest.lookup(frame_idx)` на реальный `analyzer.analyze_frame(frame)`
   (VLM) или инференс YOLO.
2. **Подключить дообученную модель.** В [`src/analyzer.py`](../src/analyzer.py)
   указать имя своей GGUF-модели в LM Studio; для YOLO — добавить отдельный
   модуль инференса.
3. **Камера → RTSP.** В `CameraStream(src=...)` указать адрес промышленной
   камеры (см. README, раздел RTSP).
4. **Асинхронность.** Тяжёлую VLM гонять в фоне (как было до манифеста); лёгкий
   YOLO — синхронно в реальном времени.

## 10. Рекомендуемый маршрут

1. **Сейчас:** собрать `s`/`d` реальные кадры → включить **Few-Shot (A)** для
   быстрой проверки на реальном продукте.
2. **Накопив ~1000 боксов:** обучить **YOLO (C)** — даст реальную real-time
   детекцию на скромном GPU.
3. **Для отчётности ОТК и редких типов:** дообучить **VLM (B)** под объяснимый
   JSON-вывод; связка `YOLO → VLM`.
4. **Промышленный GPU (A100+):** полное дообучение, высокий контекст, тысячи
   спецификаций дефектов — мгновенная классификация на полной скорости ленты.
